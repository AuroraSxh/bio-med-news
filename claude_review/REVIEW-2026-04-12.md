## Remaining Issues (Medium Priority)

### 1. Upsert Uses N+1 Query Pattern

**Location:** `backend/app/services/news_repository.py:83-117`
**Risk:** MEDIUM — Performance
**Description:** `upsert_news_items()` executes one SELECT per item in a loop. With 4 sources x 12 items = 48 queries per ingestion cycle.
**Recommendation:** Use batch lookup `WHERE canonical_url IN (...)` or PostgreSQL `ON CONFLICT` upsert.

### 2. RSS Sources Are Hardcoded

**Location:** `backend/app/services/sources.py:23-28`
**Risk:** MEDIUM — Maintainability
**Description:** Only 4 sources are hardcoded in Python. Adding/removing requires code change + redeployment.
**Recommendation:** Move to database table, env variable, or config file.

### 3. GLM5 Client Has No Retry with Backoff

**Location:** `backend/app/services/glm5_client.py:53-58`
**Risk:** MEDIUM — Reliability
**Description:** No delay between retry attempts. Transient 429/502/503 cause immediate fallback.
**Recommendation:** Add exponential backoff with `time.sleep(2 ** attempt)` for transient HTTP errors.

### 4. No Structured Logging

**Location:** All backend Python files
**Risk:** MEDIUM — Operations
**Description:** Logs use plain text format. JSON structured logs would be easier to query in production.
**Recommendation:** Add JSON log formatter.

### 5. No Monitoring / Alerting

**Location:** Infrastructure level
**Risk:** MEDIUM — Operations
**Description:** `/api/health` exists but nothing monitors it.
**Recommendation:** Add uptime monitor (UptimeRobot, Healthchecks.io, or cron-based).

### 6. SequenceMatcher Deduplication Is O(n^2)

**Location:** `backend/app/services/ingestion.py:86-89`
**Risk:** LOW at current scale (~48 items)
**Description:** Each title compared against all seen titles. Quadratic at scale.
**Recommendation:** Fine for now. Consider MinHash/SimHash if adding many sources.
